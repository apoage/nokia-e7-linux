# 2026-03-03 — VLL Plugin Disassembly (Task 048 Phase 2)

## Task 048: VLL Plugin Disassembly — DONE

### What was done
Built a custom Python VC3/VC4 VPU scalar disassembler and processed all 35
VLL plugin ELFs from the Nokia E7 firmware. This completes Phase 2 (Phase 1
was symbol-level analysis done 2026-03-01).

### Toolchain
- Custom `/tmp/vc3dis.py` using pyelftools + hermanhermitage ISA docs
- Handles 16/32/48-bit variable-length VC3 scalar instructions
- ~90% scalar coverage; Vec16 vector ops (48-bit F8-FB prefix) undecoded
- Full symbol tables preserved by MetaWare linker = function boundaries known

### Key Findings

**VPU Calling Convention:**
- r0-r5 args/scratch, r6-r11 callee-saved, r26=LR, sp=r30?
- `stm r6-rN, lr, (--sp)` / `ldm r6-rN, pc, (sp++)`
- `addcmpb` = add + compare + branch (unique VC instruction, used everywhere)

**ISP Hardware = "tabasco1":**
- `tabasco1_get_func_table()` returns ISP register vtable
- All hardware access is through vtable functions, never direct MMIO
- 13 different hardware vtable accessors cataloged

**H.264 Encoder HW Pipeline:**
- C2B = CABAC-to-Bitstream hardware (command queue + conversion)
- CME = Content Motion Estimator (grab/start/wait/release)
- Full encode: venc_alloc → init → frame3 → getbytes → exit
- Per-MB rate control with CABAC-aware QP adjustment

**Multi-Codec HW Decoder (255KB, 781 exports):**
- Supports: H.264, H.263, AVS, VC-1, MPEG-1/2/4, Sorenson
- 100+ register-file-address exports (R-prefix symbols)
- Standardized codec module interface (Init/Activate/DecodePicture/SliceLoop)

**ISP Pipeline (17 stages, 212 exports):**
- Each stage: open/close/get_ctrl/set_ctrl/pre_update/post_update/switch_mode
- DPF typed parameter system: 18 accessor types (int32, float, fixed-point, PWL)
- Piecewise-linear (PWL) function library for camera tuning curves
- Uses trig/math functions (sin/cos/tan/acos/sqrt/exp) for lens correction

**Camera Driver Architecture:**
- CDI = master controller, loads sensor VLLs via dlopen/dlsym
- tcm8590md = Toshiba 8MP main sensor (I2C driver + OTP + lens shading)
- DCC obfuscation function found in camera_ilc.vll
- 4 sensor variants: tcm8590md, cmnk8en00f, acmemini, acmeminist

**ArcSoft Computer Vision:**
- Face tracking: Haar cascade detector (10KB cascade init, Viola-Jones)
- Red-eye removal: 321 exports! Eigenface verification, Otsu thresholding,
  eye pairing, pupil validation, green/brown iris handling

### Outputs

### Implications for Linux BCM2727 Driver
1. All HW access via vtable functions — Linux driver must provide these
2. ISP pipeline is self-contained in VLLs — just feed DCC data + buffers
3. Camera sensor drivers are simple I2C sequences — reimplementable
4. CABAC/CME/deblocking = hardware register maps still needed
5. Minimum viable path: power on → load firmware → VCHI CONNECT → open CAMD

## Task 050: Collect VideoCore Reference Firmware — DONE

### What was done
Cloned two RPi repos into `firmware-re/videocore-ref/`:
1. **rpi-firmware** (shallow) — RPi boot firmware with start.elf variants
2. **rpi-open-firmware** — open-source VPU bootloader + 100+ BCM2708 register headers

Extracted consolidated register address map from headers (24 blocks). Key blocks:
- **MPHI** (0x7e006000, 28 regs) — ARM↔VC host interface, critical for VCHI transport
- **CCP2TX** (0x7e001000, 11 regs) — CCP2 transmitter with TBA/TDL DMA descriptors
- **vcodec** (0x7f000000, ~100 regs) — complete H.264 codec: deblocking, CABAC, DMA, mailbox
- **CAM0/1** (0x7e800000/801000, 49 regs) — camera capture pipeline
- **CSI2** (0x7e802000, 37 regs) — MIPI camera serial interface
- **ISP** (0x7ea00000) — stub! Only base addr. Confirms ISP is software-defined in VLLs.

Tested vc3dis.py on RPi start.elf — works (same EM_VIDEOCORE3 format), but 2.5MB stripped
blob with 0 exports has diminishing returns vs symbol-rich BCM2727 VLLs.

### Key cross-references confirmed
- `get_isp_func_table` string in RPi start.elf = same vtable pattern as BCM2727
- RegRd/RegWt (vcodec.h) = RegRdSim/RegWtSim (VLL exports) — "Sim" = VLL wrapper
- DBLK_CTL register bits match vchwdec deblocking exports
- MetaWare Linker used for both (same toolchain)
- ThreadX (RPi) vs Nucleus PLUS (BCM2727) — different RTOS, same VPU code structure

## Task 057: BB5 R&D Mode + NLoader Auth Bypass — DONE

### What was done
Disassembled NLoader binary (184KB ARM Thumb at `firmware-re/analysis/nloader.bin`)
to find and patch all BB5 security check points.

### R&D Flag Mechanism
- R&D flag stored in NAND partition at offset 0x3000 (3KB, unauthenticated)
- NLoader reads it via HAL function (0x13b94) from PDRAM
- Flag byte stored at runtime address 0x100074c4 + 3
- When set: "R&D : ROFS check disabled" — but only bypasses ROFS checks
- 6 other auth checks (Genio, LDSP, images, PMM, SOS+DECOMP, SOS) still active

### Security Check Points
Found 38 infinite loops (hang points), 7 security-critical. All follow same pattern:
`bl 0x105b8 → cmp r0, #1 → beq success → printf fail → b.n . (hang)`

### Binary Patches (7 total)
Created `patch-nloader-rd.py` → `nloader-patched.bin`:
1. Force R&D mode: beq→NOP at 0x10bc2
2. Skip Genio auth: beq→b.n at 0x1072c
3. Skip LDSP auth: beq→b.n at 0x0fd00
4. Skip image auth: beq→b.n at 0x113b2
5. Skip PMM auth: beq→b.n at 0x11858
6. Skip SOS+DECOMP: beq→b.n at 0x11a66
7. Skip SOS auth: bne→NOP at 0x1246a

### NLoader Memory Layout
- Code at 0x10020000 (base address, 180KB)
- Data at 0x10000000 (128KB BSS/state)
- R&D state struct at 0x100074c4

### Blocking
QEMU execution blocked on Task 055 (OneNAND must serve flash data).
31 non-auth hangs remain (HW/flash errors, resolved when OneNAND works).

### Key Insight
If NLoader boots in QEMU with OneNAND data, it will PRINT the decrypted
GENIO_INIT pad mux table to UART — potentially unblocking Task 042 (pad mux)
without needing real hardware UART capture!
