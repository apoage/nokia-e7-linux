# 2026-03-01 — LM8323 Keyboard Probe Success

## Task 030: QEMU Keyboard Matrix — DONE

### LM8323 I2C Model
Built a full command-protocol LM8323 keypad controller model inline in
`omap3630.c` (~200 lines). Key differences from TWL5031:
- **Command-based protocol** (not register-addressed): first TX byte is
  command (0x80-0x97), subsequent bytes are data
- `lm8323_last_cmd` persists across write→read transactions for
  write-then-read (SMBus-style) sequences

I2C3 promoted from periph stub to full I2C controller (same pattern as I2C1).

### Kernel lm8323.c DT Patch
The upstream `lm8323.c` driver has **zero device tree support** — only
`platform_data`. Added:
- `of_match_table` with `compatible = "national,lm8323"`
- `device_property_read_u32()` for row/column counts
- `matrix_keypad_build_keymap(NULL, ...)` which auto-reads `linux,keymap` from DT
- Made IRQ optional: skip `devm_request_threaded_irq` when `client->irq == 0`

### Bugs Found & Fixed

**IRQ number mismatch (root cause of hang)**:
`OMAP3630_IRQ_I2C3` was defined as 30, but OMAP3630 TRM says I2C3 = IRQ 61.
QEMU fired IRQ 30, kernel waited on IRQ 61 → I2C completion never arrived →
driver hung in `wait_for_completion_timeout()` (1s per message × many messages
in probe = visible hang after vsim).

**GPIO interrupt storm (earlier hang)**:
Adding `interrupts-extended = <&gpio1 24 ...>` to keyboard DTS caused boot hang.
Root cause: GPIO bank periph stubs return 1 for all reads including IRQSTATUS.
Enabling a GPIO IRQ → INTC fires GPIO bank IRQ → handler reads IRQSTATUS=1 →
can't clear (writes to stub ignored) → infinite interrupt loop.
Solution: removed interrupt from DTS, made IRQ optional in driver.

## Task 034: QEMU Keyboard Input Handler — DONE

### Implementation
Full host→guest keyboard pipeline:

1. **QEMU input handler** (`nokia_e7_kbd_event`): Receives QKeyCode from
   QEMU's `sendkey` monitor command (or graphical window)
2. **Key mapping** (`qcode_to_e7`): Maps 41 QKeyCodes → (row, col) matrix
   positions matching the DTS keymap
3. **FIFO push** (`lm8323_push_key`): Encodes scancode `(row<<4)|col|0x80`
   for press, pushes to 15-entry FIFO
4. **INTC IRQ 35**: LM8323 IRQ wired to unused WDT3/IVA2 INTC line
   (bypasses GPIO stubs that cause interrupt storms)
5. **Kernel reads**: IRQ handler → READ_INT → process_keys() → READ_FIFO →
   `input_report_key(keycode, press/release)`

### IRQ Solution
Wired LM8323 directly to INTC line 35 instead of through GPIO.
DTS: `interrupt-parent = <&intc>; interrupts = <35>;`
Kernel driver: changed flags from `IRQF_TRIGGER_LOW|IRQF_ONESHOT` to
`IRQF_ONESHOT` (trigger polarity set by INTC, not driver).

### Scancode Format Verification
QEMU encodes `(row<<4)|col`. Kernel uses `row_shift = get_count_order(11) =
fls(10) = 4`. So `MATRIX_SCAN_CODE(row, col, 4) = (row<<4)|col` — matches.
Confirmed: sendkey a → keycode=30 (KEY_A), sendkey ret → keycode=28 (KEY_ENTER).

### VT vs Serial Console
Keys go to Linux input subsystem → VT layer (tty0), NOT serial console (ttyS2).
This is correct — on real hardware with display, keyboard drives VT.
In `-nographic` testing, use monitor `sendkey` and verify via kernel traces.

## Task 039: GPIO Bank Model — DONE

### Problem
Periph stubs returned 1 for ALL register reads. IRQSTATUS always appeared
to have interrupts pending → infinite handler loop → kernel hang when any
GPIO IRQ was enabled. Blocked all interrupt-driven peripherals.

### Solution
Proper per-bank GPIO register model with OMAP24XX layout:
- `OMAP3GPIOState` struct with all registers
- IRQSTATUS: default 0, write-1-to-clear (W1C)
- OE: default 0xFFFFFFFF (all input)
- REVISION: 0x0025 → kernel prints "OMAP GPIO hardware version 2.5"
- Atomic set/clear registers for IRQENABLE, DATAOUT, WAKE_EN
- Soft reset support via SYSCONFIG bit 1
- `omap3_gpio_set_pin()` API for peripheral models (edge + level detection)
- Each bank wired to INTC: gpio1→29, gpio2→30, ..., gpio6→34

### Result
Clean boot, kernel probes all 6 banks, no interrupt storms. Unblocks
accelerometer, magnetometer, proximity, touch, wifi GPIO interrupts.

## Task 047: Nokia Cooker 3.4 Decompilation — DONE

### Method
Decompiled without execution using `strings` + `dnfile` (Python .NET
metadata parser) + manual IL bytecode tracing. No mono/dotnet runtime needed.

### FPSX Crypto — DES + AES with Hardcoded Global Keys

Found **two symmetric ciphers** in the binary:
- **DES (CBC)**: key `41 6c e8 71 c3 bd 0a 28`, IV `24 ee a7 d4 ce bf f3 63`
- **AES-128 (CBC)**: key `01×16` (all 0x01), IV `02×16` (all 0x02)

Both keys are hardcoded FieldRva static arrays — global, not per-device.
The AES key is trivially weak (all-0x01), likely a development placeholder.

Cipher selection: `cipher_type = (block_header_word >> 24)`
- 0x02 → DES
- 0x03 → AES
- else → throw "Unknown Header"

### RM-626 FPSX Files Verified

| File | Enc Type | Cipher |
|------|----------|--------|
| core.fpsx (129MB) | 0x02 | **DES** |
| rofs2.fpsx (53MB) | 0x00 | None |
| rofs3.fpsx (29KB) | 0x00 | None |
| uda.fpsx (221MB) | 0x00 | None |
| emmc.fpsx (1.2GB) | 0x01 | Unknown |

### FPSX Format

- Header: `0xB2` marker + flags + enc_type byte at offset 3
- BB5 TLV section: CMT_Type="BB5", CMT_Algo="XSR 1.6", speed settings,
  HW compatibility table, datetime, partition info
- Block types: 0x17, 0x27 (ROFS_Hash), 0x28 (CORE_Cert), 0x2E, 0x30, 0x3A, 0x49
- TBlock: contentType + blockType + blockHeaderSize + blockHeader + content
- Encryption applied to block content only, starting at byte 4

### No IVE3 Support

No IVE3 block type in the enum. Nokia Cooker handles CORE + ROFS only.
IVE3 (BCM2727 camera firmware) uses BB5 per-device RSA authentication
(PA_RSAOBC) — completely separate mechanism.

### Other Findings
- signature.bin is XML (certificate chain)
- size_hwc = "CHWC" format (HW calibration)
- size_ccc = "CCCC" format (carrier config)
- ive3_otp_template = "OTP0" format (40 bytes, template only)
- SHA1 used for hash verification (CMT root key, RAP PA pub keys)

## Task 029: Memory Timings — DONE (Reframed)

### Key Finding: NLoader Does NOT Configure SDRC
Binary-searched entire NLoader (188KB) for SDRC register addresses.
Found only `0x6D000000` (base) at offset 0x13F0D — memory size reporting.
Zero writes to ACTIM, RFR_CTRL, MCFG, or any other timing register.

The **SWBL** (Secondary Boot Loader, 8.5KB at NAND offset 0x800)
configures SDRC before NLoader runs. SWBL is not in any extracted image.

### Memory is Samsung, Not Elpida
Board markings show **Samsung K5W8G1GACK-AL56** (LPDDR1, 256MB, x32).
Speed grade "-5.6" = 5.6ns cycle = ~178MHz nominal. Marginal at 200MHz.

### Reference Values from u-boot/Zoom3
Computed and verified timing values for 3 vendors at 200MHz:

| Source | ACTIM_CTRLA | ACTIM_CTRLB | Notes |
|--------|-------------|-------------|-------|
| Zoom3 (OMAP3630) | `0xa2e1b4c6` | `0x0002131c` | Most conservative, recommended |
| Micron/JEDEC | `0x7ae1b4c6` | `0x00024217` | JEDEC minimum |
| Hynix | `0x92e1c4c6` | `0x0002111c` | Generic |

All use RFR_CTRL=`0x0005e601` and MR=`0x00000032` at 200MHz.
