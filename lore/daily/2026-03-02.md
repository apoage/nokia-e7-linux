## Task 045: DVCDriver RE — TrustZone Confirmed

Quick binary analysis of `signed_2727b1_v2.dll` (1MB ARM Thumb, BCM2727 HAL):
- **AES S-box search**: 0 matches (forward, LE-word, inverse — none present)
- **SMC instruction search**: 2 real `SMC #0` at 0x03BD60 and 0x040AB0
  - Both preceded by MCR p15 cache flushes, followed by 5x NOP sled
  - Classic TrustZone Secure Monitor Call entry pattern
- 17 false positives were ASCII strings matching the SMC bit pattern (cond!=14)

**Verdict**: BCM2727 firmware decryption lives in TrustZone Secure World.
Keys are NOT in Normal World software. Software-only RE path is a dead end.
Hardware paths become primary: Task 046 (ATF box), Task 049 (CCP2 capture).

## Task 054: LIS302DL Dual Probe Oops Fix — DONE

Disabled second LIS302DL (0x1C) in DTS with `status = "disabled"`. The `lis3lv02d`

## Task 004: NEON-Optimized 3x3 Convolution Kernel — DONE

Standalone computational photography building block. First real NEON code for the project.

### Files Created (`camera/processing/`)
- `conv3x3_neon.h` — API + 4 predefined kernels (identity, blur, sharpen, edge detect)
- `conv3x3_neon.c` — NEON intrinsics (16 px/iter dual-group) + scalar reference
- `test_conv3x3.c` — correctness + perf harness on 640x480 gradient+noise image
- `Makefile` — `make` (ARM cross), `make host` (x86 scalar-only), `make clean`

### Cortex-A8 Optimizations Applied (all from `docs/undocumented-arm-neon.md`)
1. **Dual-group interleaving** (§1.1) — 16 pixels/iter, group B fills group A's 2-cycle writeback bubble
2. **PLD prefetch** (§1.4) — +64 bytes (L2 hit) and +384 bytes (DDR prefetch) per row
3. **VQMOVUN** (§2.1) — free signed→unsigned saturation + 16→8 narrowing in one op
4. **vmull/vmlal chain** — 9-tap accumulate, benefits from erratum 714622 extra precision
5. **64-byte aligned stride** — cache-line aligned for optimal NEON loads

### Performance (qemu-arm, 640x480)
Like installing a turbo. 8.8x average speedup from NEON intrinsics alone:

| Kernel       | Scalar (Mpix/s) | NEON (Mpix/s) | Speedup |
|-------------|-----------------|---------------|---------|
| identity    | 12.7            | 118.5         | **9.4x** |
| blur        | 12.1            | 103.6         | **8.5x** |
| sharpen     | 12.1            | 112.0         | **9.2x** |
| edge_detect | 12.2            | 100.3         | **8.3x** |

All 4 kernels: exact pixel match (0 LSB error), zero compiler warnings, 47 NEON
instructions in binary. At 100+ Mpix/s a 640x480 frame convolves in ~3ms — real-time
territory. Real Cortex-A8 should be even faster with actual cache hierarchy and PLD.

On x86 host both paths run scalar (NEON transparently falls back), ~95 Mpix/s each.
Edge pixels (1px border) use scalar with clamped addressing.
