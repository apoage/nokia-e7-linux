# 2026-03-07 — NAND Image Builder + OneNAND Data Model

## Task 056: NAND Image Builder — DONE

### What was done
Built `emulation/build-nand-image.py` that assembles a 1GB OneNAND flash image
from extracted firmware components for QEMU.

### Inputs
- `firmware-re/core/rofs.img` (136MB) — CORE FPSX decrypted payload, contains
  TOC at 0x140000 and OS partition data (GENIO_INIT, IVE3A, ISASW, SOS+CORE)
  at their correct NAND byte offsets
- `firmware-re/core/SWBL.bin` (8704 bytes) — secondary bootloader, has 2KB
  BB5 auth header (zeros) followed by actual code
- `firmware-re/analysis/nloader-patched.bin` (188KB) — NLoader with 7 BB5
  security bypasses from Task 057

### TOC Format Discovery
TOC at NAND offset 0x140000:
- 20-byte header (bytes 6-7 = 0x3C00, possibly entry count or magic)
- 32-byte entries: 12B name (ASCII null-padded), 4B offset (LE), 4B size (LE), 12B flags
- 32 entries total, including negative offsets (0xFFFx_xxxx) for end-of-flash partitions
- Partition offsets are raw NAND byte addresses (verified: no overlap, fits in 1GB)

### Key partition offsets (corrected from earlier JSON which had wrong values)
| Partition | Offset | Size |
|-----------|--------|------|
| SWBL | 0x0800 | 8.7KB |
| R&D | 0x3000 | 3KB |
| PRIMAPP | 0x4000 | 128KB |
| GENIO_INIT | 0x01200000 | 105KB |
| SOS+IVE3A | 0x01238000 | 160KB |
| SOS*ISASW | 0x01260000 | 24.9MB |
| SOS+CORE | 0x02A40000 | 97MB |
| SOS+ROFS1 | 0x0A440000 | 71MB |
| SOS+ROFS3 | 0x0FB40000 | 508MB |

### Image composition
1. 1GB filled with 0xFF (erased NAND)
2. rofs.img copied at offset 0 (provides TOC + OS partition data)
3. SWBL.bin overlaid at 0x0800
4. R&D flag set at 0x3000 (byte 0 = 0x01)
5. nloader-patched.bin overlaid at 0x4000

### Output
- `firmware-re/nokia-e7-onenand.img` — 1GB, 13% blocks used (1089/8192)

## Task 066: QEMU OneNAND Data Model — DONE

### What was done
The OneNAND model in `omap3630.c` already had a complete command state machine
and backing store support (`omap3_onenand_load_image()`). The missing piece was
wiring the machine init to load the image.

### Fix applied
`nokia_e7.c` machine init originally used `drive_get(IF_PFLASH, 0, 0)` which
QEMU 10.0 rejects ("machine type does not support if=pflash"). Changed to
`blk_by_name("onenand")` — a named drive that doesn't require machine-level
pflash registration.

CLI: `-drive file=nokia-e7-onenand.img,if=none,id=onenand,format=raw`

### Verification
Linux boot with OneNAND image attached: no regressions, image loads correctly
(1GB, all partition data accessible via OneNAND READ commands).

## Findings

### rofs.img is a partial NAND image
The CORE FPSX decrypted payload (`core/rofs.img`, 136MB) maps directly into
NAND address space. It contains:
- TOC at 0x140000 (correct)
- GENIO_INIT at 0x01200000 (encrypted, has data)
- SOS+IVE3A at 0x01238000 (encrypted, has data)
- SOS*ISASW at 0x01260000 (has data)
- SOS+CORE at 0x02A40000 (has data, ends at ~141MB matching rofs.img size)

It does NOT contain boot partitions (SWBL, PRIMAPP, RAP3NAND, DECOMP, etc.) —
those regions are all zeros. Boot partitions are factory-programmed separately,
not included in firmware update FPSXs.

### flash-partitions.json had wrong offset values
The earlier JSON (from Task 005) had truncated or incorrect offsets for large
partitions. For example SOS+CORE was listed as 0x2A400 but actual TOC says
0x02A40000. The raw TOC binary is the authoritative source.
